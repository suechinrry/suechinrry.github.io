创建仓库

  • 使用git init

    • 会在当前目录下创建一个 .git 子目录，里面包含了 Git 所需的所有文件和数据结构，用于跟踪版本历史。

      此时，你可以开始在这个目录下使用 Git 来管理文件，进行版本控制等操作。

    • 后面跟名称会在当前目录下新建名称为已给名称的目录

  • 使用git clone

    补充：

    点击 Code 按钮，你会看到一个 URL 地址框。

    • HTTPS URL：如果你使用 HTTPS 协议，复制 https://github.com/用户名/仓库名.git。
    • SSH URL：如果你使用 SSH 协议，复制 git@github.com:用户名/仓库名.git（前提是你已经配置了 SSH 密钥）。
    1

    git clone https://github.com/用户名/仓库名.git

将文件添加到仓库

  • git status 查看仓库状态

追踪了并且提交到本地仓库的、且未作修改的文件，在执行 git status 时 不会显示。

这是因为 git status 只会显示以下几种情况：

  1. 修改了的文件（即已经追踪的文件有更改，但尚未提交到仓库）。
  2. 新增文件（尚未被 Git 跟踪的新文件）。
  3. 已暂存的文件（已用 git add 添加到暂存区，准备提交的文件）。
  4. 与远程仓库的差异（如有未推送的提交或与远程分支不同的状态）。

  • git add 将文件添加到暂存区

    • 可以使用通配符批量添加

    • 还可以使用git add . 添加当前文件夹下的所有文件

    • git rm --cached <文件名> 从暂存区移除

  • git ls-files 查看暂存区里的内容

  • git commit 提交暂存区里的文件到本地仓库

    正确的使用方式（包含提交信息） git commit -m "修复登录页面的样式问题" ``-m ，选项用于为提交添加一条提交信息

    提交信息为空，Git 会报错

  • git log 查看提交记录

    后面可以跟选项，展示提交记录

回退版本

默认为mixed

补充：提交后，暂存区的内容不会被删除，暂存区依然可以用来存放对文件的新修改，直到再次提交。

git reset --XX 提交id（版本id）

  • git reset --soft

    回退到某个版本，但是保留工作区和暂存区的所有内容

  • git reset --hard

    回退到某个版本，但是丢弃工作区和暂存区的所有内容

  • git reset --mixed

    回退到某个版本，但是保留工作区的内容

查看差异

git diff

1. 查看工作目录和暂存区之间的差异

git diff 默认情况下，比较工作目录和暂存区的差异，也就是你已经修改但尚未使用 git add 暂存的文件内容：

1

git diff

它会显示自上次提交以来，工作目录中尚未暂存的更改。

2. 查看暂存区和上次提交之间的差异

如果你已经通过 git add 将更改暂存，但还没有提交，可以使用以下命令查看暂存区和最后一次提交之间的差异：

1

git diff --cached

这个命令会显示暂存区的内容和上一次提交的内容之间的差异。

3. 查看工作目录和上次提交之间的差异

HEAD是分支的最新提交节点

如果你想查看工作目录和最后一次提交之间的所有差异（包括已经暂存和未暂存的更改），可以使用：

1

git diff HEAD

这里的 HEAD 表示当前分支的最新提交。

4. 查看某个文件的差异

你也可以指定某个文件来查看它的差异。例如，要查看 file.txt 的差异，可以使用：

1

git diff file.txt

这样只会显示 file.txt 文件的更改。

5. 查看两个分支之间的差异

如果你想查看两个分支之间的差异，可以指定两个分支名。例如，查看 main 和 feature 分支之间的差异：

1

git diff main..feature

这会显示 feature 分支相对于 main 分支的差异。

6. 查看两个提交之间的差异

如果你想查看两个特定提交之间的差异，可以使用：

1

git diff commit1 commit2

其中 commit1 和 commit2 是提交的 SHA-1 哈希值、分支名或者引用（例如 HEAD~1）。

7. 查看某个文件在两个提交之间的差异

如果你只对某个文件在两个提交之间的差异感兴趣，可以使用：

1

git diff commit1 commit2 -- file.txt

8. 查看修改的统计信息

如果你只想查看文件修改的统计信息，而不显示具体的修改内容，可以加上 --stat 选项：

1

git diff --stat

删除文件

git rm 文件名

  • --cached 只从 Git 跟踪中移除文件，保留文件在本地磁盘上。
  • -r 或 --recursive 递归删除目录及其内容。
  • -f 或 --force 强制删除文件，即使文件是只读的或有未提交的修改。
  • -n 或 --dry-run 模拟删除操作，显示将被删除的文件，但不实际删除。

删除工作区和暂存区，但是版本库中还存在，所以还要记得提交

.gitignore文件

生效前提：要忽略的文件不能是已经添加到仓库里的文件

文件格式

.gitignore 文件的格式非常简单，每行代表一个模式，用来匹配需要忽略的文件或目录。以下是 .gitignore 文件的一些常见规则：

常见规则和语法

使用标准的Blob模式匹配

  • \*：匹配零个或多个字符。

    例子：*.txt 匹配所有 .txt 文件。

  • ?：匹配单个字符。

    例子：file?.txt 匹配 file1.txt, fileA.txt 等，但不匹配 file10.txt。

  • []：匹配指定范围内的单个字符。

    例子：file[1-3].txt 匹配 file1.txt, file2.txt, file3.txt。

  • \**（用于递归匹配目录）：

    例子：**/*.txt 匹配所有子目录中的 .txt 文件。

  1. 忽略特定文件或目录

    • 你可以指定要忽略的文件或目录的路径。例如：

    • 忽略文件夹必须要带/

      1
2
3

      example.txt
logs/
build/

    • 上面表示 example.txt 文件，logs 目录和 build 目录会被 Git 忽略。

  2. 忽略特定扩展名的文件

    • 使用通配符 * 来忽略所有符合模式的文件。例如：

      1
2

      *.log
*.tmp

    • 这样会忽略所有 .log 和 .tmp 后缀的文件。

  3. 忽略某个文件夹中的某种类型文件

    • 如果你只想忽略特定文件夹中的文件，可以指定路径。例如：

      1

      logs/*.log

    • 这表示只忽略 logs 文件夹中的 .log 文件。

  4. 忽略目录中的所有文件

    • 要忽略某个目录中的所有文件，使用 * 通配符。例如：

      1

      build/*

    • 这会忽略 build 目录中的所有文件，但 build 目录本身不会被忽略。

  5. 取消对已忽略文件的忽略

    • 如果 .gitignore 中的某个文件之前被忽略，但你后来希望 Git 追踪它，可以使用 ! 取消忽略：

      1

      !important.txt

    • 上面的规则表示取消对 important.txt 文件的忽略，Git 会开始追踪它。

  6. 注释

    • 你可以在

      1

      .gitignore

      文件中添加注释，注释以 # 开头。例如：

      1
2

      # 忽略所有的 .log 文件
*.log
